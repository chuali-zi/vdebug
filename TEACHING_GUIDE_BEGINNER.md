# LOT 新手教学指导

这份文档不是给“已经会做项目的人”写的，而是给第一次接触这种项目形态的你写的。

你的困惑很正常，因为这个仓库表面上看起来像“我在做一个调试工具”，但代码实现却是：

- 不是传统命令行一步一步跑完就退出的脚本
- 也不是单纯点按钮的网页
- 而是一个“先启动服务，再通过接口和它交互”的后端程序

如果这几个概念混在一起，大一学生很容易直接乱掉。下面我按“先建立地图，再看细节”的顺序讲。

## 1. 先讲清楚：这个项目到底是什么

这个项目当前本质上是一个：

- 嵌入式调试场景的模拟后端
- 用 HTTP API 对外提供能力
- 运行在本地进程里
- 内部再拆成多个模块协作

简单说，就是：

1. 你先把程序启动起来
2. 它开始监听某个端口，比如 `8000`
3. 你再通过请求告诉它“创建一个 session”“推进 10ms”“执行一次 i2c 事务”“跑一个 scenario”
4. 它把结果用 JSON 返回给你

所以它现在更像一个“小型本地服务”，不是“传统一次性 CLI 工具”。

## 2. 你为什么会觉得乱：CLI 思维和 HTTP 服务思维不一样

### 2.1 传统 CLI 是线性的

你以前更容易理解的可能是这种程序：

```text
输入命令 -> 程序运行 -> 打印结果 -> 程序退出
```

比如：

```bash
python tool.py input.txt
```

这种程序通常是一条线：

1. 读输入
2. 处理
3. 输出
4. 结束

这叫“单次执行模型”。

### 2.2 HTTP 服务不是单次执行模型

这个仓库更像这样：

```text
先启动服务
    |
    v
服务一直活着，等别人发请求
    |
    v
每来一个请求，就处理一次
    |
    v
返回 JSON，但服务本身不退出
```

也就是说，它不是“你运行一次，做完一件事就结束”。

而是：

- 服务先常驻
- 状态可以保存在内存或文件里
- 多次请求一起构成一次完整流程

这就是你感受到“不像正常线性逻辑”的根本原因。

## 3. 那它到底算 CLI 吗

严格说，当前仓库主实现不是 CLI，本体是 HTTP 服务。

但是它可以被 CLI 包起来。

### 3.1 当前仓库里真正的核心是 HTTP 服务

服务入口在：

- [main.py](D:/projects2/Lot/src/lot/main.py)

推荐启动方式：

```bash
python -m uvicorn --app-dir src lot.main:app --reload --host 0.0.0.0 --port 8000
```

启动之后，它就变成一个本地服务。

### 3.2 那 `scripts/run.ps1` 和 `scripts/run.sh` 是什么

它们不是业务核心，只是“启动脚本”。

你可以把它理解成：

- 帮你切到仓库根目录
- 帮你激活虚拟环境
- 帮你补 `PYTHONPATH`
- 最后替你执行 `python -m uvicorn ...`

也就是说，这两个脚本只是“方便启动服务”的壳，不是系统本身。

### 3.3 如果以后你真要做 CLI，该怎么理解

以后你可以在这个 HTTP 服务外面再包一层 CLI，比如：

```bash
lot-cli create-session
lot-cli step --session sess-123 --delta 10
lot-cli run-scenario --session sess-123 --file scenarios/example_i2c_stuck.yaml
```

这个 CLI 内部做的事，本质上可能只是：

1. 读命令行参数
2. 发 HTTP 请求给本地服务
3. 把返回的 JSON 打印得更友好一点

所以：

- HTTP 服务是“发动机”
- CLI 可以是“驾驶台”

你现在乱，是因为你以为自己在做“驾驶台”，但代码主体其实先把“发动机”搭出来了。

## 4. 这个项目最重要的数据流

先记住一句话：

> 这个项目的主线不是“写一个函数算结果”，而是“把一次调试过程拆成多个模块接力”。

主链路可以先粗看成：

```text
请求 -> API -> Session -> Board -> Engine -> Devices -> Diagnosis -> Artifacts -> 返回结果
```

下面我把每一站讲清楚。

### 4.1 API 模块在做什么

它相当于前台接待。

作用：

- 定义有哪些 HTTP 路由
- 接住外部请求
- 校验请求参数
- 调用后面的服务
- 把结果包装成统一 JSON

它不负责复杂业务，它负责“接入”和“转发”。

### 4.2 Session 模块在做什么

它相当于“这次调试任务的容器”。

一个 session 可以理解成：

- 一次运行
- 一个上下文
- 当前板卡、时间、状态、事件、诊断都挂在它上面

为什么需要它？

因为 HTTP 请求是一条一条来的，如果没有 session，服务就不知道“这次 step 是接着哪个运行继续做”。

### 4.3 Board 模块在做什么

它负责读板卡配置。

比如：

- 有哪些总线
- GPIO 怎么配
- I2C 有哪些设备
- UART 波特率是多少

来源通常是：

- `profiles/*.yaml`

所以 `board` 的职责不是“跑模拟”，而是“把硬件描述读进来并校验”。

### 4.4 Engine 模块在做什么

它是系统里的调度中心。

你可以把它理解成：

- 负责虚拟时间
- 负责 step
- 负责调度未来事件
- 负责把 `gpio:set`、`uart:send`、`i2c:transact` 这样的动作派发下去

它有点像“总控制器”。

### 4.5 Devices 模块在做什么

它负责每种总线或设备的具体行为。

比如：

- GPIO 写高低电平
- UART 发字节
- I2C 发事务、处理 NACK、故障注入

这里比较像“真正干活的人”。

Engine 决定“该做什么”，Devices 负责“具体怎么做”。

### 4.6 Diagnosis 模块在做什么

它负责把事件变成“更像老师能读懂的话”。

链条是：

```text
事件 event -> 诊断事实 fact -> 解释 explanation
```

为什么要分三层？

因为直接从原始事件跳到结论容易瞎猜。

举例：

1. 先产生事件：`I2C_BUS_STUCK_LOW`
2. 再抽成事实：`bus_stuck_low`
3. 再给解释：可能是 SDA 被拉低、上拉有问题、设备占线

这比“看到异常就直接乱写结论”要严谨很多。

### 4.7 Scenario 模块在做什么

它负责“场景回放”。

你可以把它理解成：

- 读一个 YAML 场景文件
- 按时间执行一系列动作
- 最后做断言

例如：

1. 第 0ms 注入故障
2. 第 1ms 发起一次 I2C 事务
3. 检查有没有出现某种诊断

这就像写一个“小实验脚本”。

### 4.8 Artifacts 模块在做什么

它负责保存证据。

包括：

- 事件流
- 诊断事实
- 解释
- 状态快照
- repro bundle

这一步很重要，因为没有产物，你就没法复盘“刚才到底发生了什么”。

## 5. 这几个模块接起来，到底发生了什么

下面我用“跑一个 scenario”为例，带你看完整流程。

### 5.1 第一步：先创建 session

你发一个请求：

```text
POST /v1/sessions
```

它会：

- 读取 board profile
- 创建 session
- 初始化 runtime

### 5.2 第二步：运行 scenario

你再发一个请求：

```text
POST /v1/sessions/{id}/scenario:run
```

它会：

1. 读取 scenario YAML
2. 按时间顺序解释其中的动作
3. 调用 engine 推进时间
4. engine 再调用 devices 执行动作
5. 产生事件
6. diagnosis 把事件变成事实和解释
7. artifacts 把这些东西落盘
8. scenario 检查断言是 pass 还是 fail
9. API 把结果打包成 JSON 返回

所以这里不是单函数，而是一条“模块流水线”。

## 6. 根目录里，除了 `src` 和 `report`，其他文件夹都是什么

这是你现在最需要的一张地图。

### 6.1 `.git`

Git 仓库元数据。

作用：

- 保存版本历史
- 记录分支、提交、索引

你一般不手动改它里面的内容。

### 6.2 `.venv`

Python 虚拟环境目录。

作用：

- 隔离这个项目的依赖
- 避免污染你全局 Python 环境

你可以理解成“这个项目自己的 Python 小世界”。

### 6.3 `profiles`

板卡配置样例目录。

里面放的是：

- 板卡 YAML
- 总线/GPIO/设备定义

这是 `board` 模块要读的输入之一。

### 6.4 `scenarios`

场景脚本目录。

里面放的是：

- 场景 YAML
- 用于回放和断言的测试流程

这是 `scenario` 模块要读的输入之一。

### 6.5 `scripts`

启动辅助脚本。

里面现在有：

- `run.ps1`
- `run.sh`

它们的作用不是实现业务，而是帮你更方便地启动服务。

### 6.6 `tests`

测试目录。

这里放的是自动化测试代码。

作用：

- 检查接口是不是还正常
- 检查引擎、设备、场景等模块有没有回归
- 帮你在改代码后快速验证

### 6.7 `runtime_sessions`

运行时 session 数据目录。

服务执行过程中，会把一些 session 相关信息落在这里。

你可以理解为：

- “这次运行的上下文存档”

### 6.8 `runtime_artifacts`

运行时产物目录。

会保存：

- event stream
- facts
- explanations
- snapshot
- bundle

你可以理解成“调试证据输出目录”。

### 6.9 `runtime_probe`

这个目录通常表示运行时探针/临时验证产物。

在当前仓库里，它更像实验或调试过程中留下的辅助目录，不是核心源代码目录。

### 6.10 `.tmp`

临时目录。

通常是：

- 调试时产生的临时文件
- 某些脚本运行时留下的缓存

### 6.11 `.test_tmp`

测试专用临时目录。

测试运行时会在这里创建临时文件夹，结束后再删除。

测试辅助代码在：

- [support.py](D:/projects2/Lot/tests/support.py)

它会：

- 在 `.test_tmp` 下创建临时子目录
- 测试结束时自动清理

### 6.12 `.test_probe`

这个目录和 `.test_tmp` 类似，通常也是测试或探测用途的临时目录。

对你现在学习来说，把它理解成“测试留下的辅助目录”就够了。

## 8. 测试到底是怎么测的

这是非常关键的一部分。

很多初学者以为测试就是“我手动运行一下看看行不行”，但自动化测试不是这个逻辑。

自动化测试本质上是在做：

1. 先准备一个输入
2. 调用你的代码
3. 检查结果是否符合预期
4. 不符合就报错

## 9. 这个项目里的测试分几类

### 9.1 API 测试

例如：

- `tests/test_api.py`

它会检查：

- 路由能不能响应
- 返回 JSON 格式对不对
- 错误时是不是统一错误结构

这里有个很重要的点：

它不是非得真的开一个浏览器去访问。

它可以直接在 Python 里模拟 ASGI 请求，把请求送进 FastAPI app。

这叫“程序内测试接口”。

### 9.2 模块测试

例如：

- `tests/test_engine.py`
- `tests/test_devices.py`
- `tests/test_board_service.py`

这些测试不会经过完整 HTTP。

它们会直接测试某个模块：

- 给它一个输入
- 看输出是不是对
- 看状态有没有正确变化

这种测试更像“单元测试”或“模块测试”。

### 9.3 场景回归测试

例如：

- `tests/test_scenario.py`

它会重点看：

- 场景 DSL 能不能正确运行
- assertion 是否正确判断
- bundle 是否导出

这更接近“小型集成测试”。

## 10. 测试为什么能说明成功

因为测试里写了“应该发生什么”。

比如：

- 期待返回状态码 `200`
- 期待 JSON 里 `ok == true`
- 期待出现某个 event type
- 期待某个断言结果是 `pass`

如果代码实际运行结果和这些期待一致，测试就通过。

如果不一致，测试就失败。

所以“测试成功”的含义不是“我感觉它好像没问题”，而是：

> 代码满足了测试里明确写下来的预期。

## 11. 这个项目里，测试成功的标准是什么

运行命令：

```bash
python -m unittest discover -s tests -p "test_*.py" -v
```

如果最后看到类似：

```text
Ran 27 tests in ...

OK
```

说明：

- 被发现的 27 个测试都通过了
- 没有失败
- 没有异常

这就是当前仓库里“测试成功”的正式信号。

注意，这不代表“项目已经完美”，只代表：

- 当前这些测试覆盖到的功能是对的

测试永远只能证明“测到的部分没出错”，不能证明“所有东西都绝对没问题”。

## 12. 你现在最该建立的认知

你现在最重要的不是死记每个函数，而是先建立这四个判断：

### 12.1 这是一个服务，不是一次性脚本

所以你要习惯：

- 启动服务
- 发请求
- 保持状态
- 多次交互

### 12.2 模块之间是接力，不是乱写在一起

你应该总问自己：

- 这个逻辑属于哪个模块
- 输入从哪来
- 输出往哪去

### 12.3 YAML 是输入，JSON 是输出

在这个仓库里：

- `profiles/` 和 `scenarios/` 里的 YAML 更像“配置/实验脚本”
- HTTP 返回的 JSON 更像“程序结果”

### 12.4 测试是“写下预期”，不是“靠感觉检查”

这是你从“会写代码”走向“会做工程”的关键一步。

## 13. 给你的学习建议

如果你现在压力很大，不要试图一次全懂。建议按这个顺序学：

1. 先只学“如何启动服务”和“如何访问 `/docs`”
2. 再学 `POST /v1/sessions` 是怎么创建 session 的
3. 再学 `step` 和 `io` 是怎么走到 engine/devices 的
4. 再学 scenario 为什么像“实验脚本”
5. 最后再学 diagnosis 和 artifacts

你现在最不需要做的事，是一上来把所有模块源码从头读到尾。

那样很容易信息过载。
